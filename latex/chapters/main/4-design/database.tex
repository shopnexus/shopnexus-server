\section{Thiết kế cơ sở dữ liệu}
\label{sec:database}

Module Order sử dụng schema \texttt{order.*} trên PostgreSQL~18, gồm 7 bảng và 4 kiểu enum, chuẩn hoá 3NF. Thiết kế tuân theo mô hình checkout hai pha: item bắt đầu ở trạng thái chờ xác nhận (\texttt{order\_id IS NULL}), chỉ được gắn vào đơn hàng sau khi seller xác nhận. Không có foreign key liên schema --- tính toàn vẹn liên module được đảm bảo ở tầng nghiệp vụ.

\subsection{Ràng buộc và chỉ mục}
\label{subsec:db-constraints}

\begin{itemize}[nosep]
    \item \textbf{Enum dùng chung:} \texttt{order.status} (Pending, Processing, Success, Cancelled, Failed) được dùng cho payment, order, refund, refund\_dispute.
    \item \textbf{Partial index:} \texttt{item\_pending\_seller\_idx} và \texttt{item\_pending\_account\_idx} lọc \texttt{WHERE order\_id IS NULL} --- tối ưu truy vấn inbox mục chờ.
    \item \textbf{FK cascade:} \texttt{item.order\_id} $\to$ \texttt{order.id} (ON DELETE CASCADE); \texttt{refund.order\_id} $\to$ \texttt{order.id} (ON DELETE NO ACTION --- giữ lại refund record).
    \item \textbf{FK set null:} \texttt{order.payment\_id}, \texttt{order.transport\_id} (ON DELETE SET NULL).
    \item \textbf{Unique:} \texttt{cart\_item(account\_id, sku\_id)} --- mỗi tài khoản chỉ có 1 dòng/SKU trong giỏ.
\end{itemize}

\subsection{Truy vấn dữ liệu (SQLC queries thay cho Stored Procedure)}
\label{subsec:db-sqlc}

ShopNexus không dùng Stored Procedure (SP) truyền thống. Thay vào đó, toàn bộ truy vấn được quản lý bằng \textbf{SQLC} --- công cụ sinh code Go type-safe trực tiếp từ SQL thuần. Lý do lựa chọn:

\begin{itemize}[nosep]
    \item \textbf{Code-first, version control}: file \texttt{.sql} nằm cùng repo, được review qua Pull Request như mọi thay đổi code khác.
    \item \textbf{Type-safe}: SQLC sinh Go struct khớp schema, lỗi được phát hiện ở compile time thay vì runtime như SP.
    \item \textbf{CI/CD}: deploy queries cùng code qua \texttt{make generate}, không cần migration riêng cho SP.
    \item \textbf{Cùng mục tiêu đóng gói}: tầng \texttt{db/sqlc/*.go} đóng vai trò Repository/DAO --- code nghiệp vụ chỉ gọi hàm Go, không viết SQL trực tiếp; DB Admin có thể điều chỉnh cấu trúc bảng, chỉ cần regenerate SQLC code.
    \item \textbf{Trade-off}: mất một số tính năng SP (cached execution plan, DB-level permission). Bù lại: đơn giản, testable, phù hợp workflow Go.
\end{itemize}

\subsection{Ràng buộc toàn vẹn dữ liệu (DB constraints)}
\label{subsec:db-integrity}

ShopNexus không dùng TRIGGER. Thay vào đó, toàn vẹn dữ liệu được đảm bảo qua ba cơ chế phối hợp:

\begin{enumerate}[nosep]
    \item \textbf{DB-level constraints} (FK, UNIQUE, enum, partial index) --- nêu trong phần này.
    \item \textbf{Business rules ở tầng biz} --- xem phần~\ref{subsec:db-biz-rules}.
    \item \textbf{Durable execution qua Restate} --- đảm bảo exactly-once cho side effects.
\end{enumerate}

Lý do chọn constraint thay trigger:

\begin{itemize}[nosep]
    \item \textbf{Auditable}: constraint nằm trong migration file, review được qua Pull Request.
    \item \textbf{Transactional}: tự động bảo vệ trong transaction, không race condition.
    \item \textbf{Debug dễ}: lỗi constraint violation rõ ràng, không ẩn sau trigger logic.
    \item \textbf{Tránh lock contention}: trigger có thể gây lock chéo và khó viết test đơn vị.
\end{itemize}

\subsection{Ràng buộc nghiệp vụ ở tầng Biz (thay cho Trigger)}
\label{subsec:db-biz-rules}

Các ràng buộc không thể biểu diễn qua constraint DB (state transition rules, cross-table invariants, gọi API bên ngoài) được thực thi ở tầng biz (Go handler). Đặc điểm nổi bật: rule chạy bên trong \texttt{restate.Run()} --- durable, idempotent, retry tự động khi có lỗi transient.

Lý do chọn biz rules thay trigger:

\begin{itemize}[nosep]
    \item \textbf{Testable}: unit test rule độc lập với DB state, không cần spin up database.
    \item \textbf{Rõ ràng}: logic trong Go dễ đọc/review hơn PL/pgSQL trigger.
    \item \textbf{Flexibility}: rule phức tạp (gọi API bên ngoài, cross-module) không thể viết trong trigger.
    \item \textbf{Observable}: log, metric, trace đầy đủ qua tầng biz; trigger thường không có.
\end{itemize}

\subsection{Tiêu chí kiểm thử thiết kế}
\label{subsec:design-test}

Các tiêu chí kiểm thử thiết kế được phân thành bốn nhóm, mỗi nhóm tương ứng với một góc nhìn chất lượng khác nhau: (A)~cấu trúc và chuẩn hoá, (B)~độ tin cậy và tương tranh, (C)~bảo mật và tuân thủ, (D)~hiệu năng và khả năng quan sát. Phương pháp V\&V (Verification \& Validation) được chọn phù hợp với bản chất từng tiêu chí --- từ rà soát ngang hàng (peer review), thanh tra chính thức (Fagan inspection), đến phân tích tự động (static analysis) và mô hình hoá (model checking).
